Metrics Forwarding Library

The metrics forwarding library sends Go runtime and custom metrics from a service to a UDP port using the StatsD format.

Existing Implementations

There were two services that forwarded metrics to Telegraf, an agent that collects, processes, aggregates, and writes metrics using a variety of plugins, which then outputted the metrics to an InfluxDB instance. Those services implemented the collection and forwarding logic differently, and the team was in the process of developing more Go services. So, they wanted to pull the metrics collecting and forwarding logic into a separate Go library that could be used by the Go services.

Some examples of metrics are number of goroutines, CPU and memory for Kubernetes pods, and API endpoint response times.

To design and develop the metrics library, I needed to analyze the current implementations to identify their features and understand how they worked. Some key features of the two implementations are

  • Collect Go runtime metrics at a specified time interval
  • Cache the metrics and then push batches of metrics to Telegraf
  • Collect custom metrics, like request response times

In terms of design, both implementations had interfaces for a sink to collect metrics and a pusher to forward metrics.

Telegraf

We use Telegraf to move metrics from our Go services to an InfluxDB instance. We start a Telegraf instance that we can access through a UDP port, and we configure it to output metrics to InfluxDB, which we pull timeseries data from for a few Grafana dashboards.

The metrics library does not setup a Telegraf instance. Instead, the Go services do, allowing for more flexibility in how the metrics library is used. The metrics library only takes in a UDP port to forward to, and the services can configure the metrics library to use Telegraf's UDP port.

Design

The metrics library combines parts of both implementations. There is a globals.go file, which contains references to the sink and pusher to use. There are also pusher.go and sink.go files containing Sink and Pusher interface definitions and implementations and some collection-related functions.

Testing and Debugging

When implementing the design, I used test-driven development to avoid adding extra code and ensure that everything was working correctly. I also mocked Go objects. My only previous mocking experience was with EasyMock, and the Go mocking packages seemed complex. So, I made simpler mocks that tested arguments and return values.

I had many opportunities to apply what I learned in a debugging course, and those debugging strategies seemed to make me faster at debugging. Some of the most valuable strategies I used were writing down what I tried during the debugging process and the results to remember what I had tried and splitting up the system and testing smaller parts to narrow down the area with the defect.
