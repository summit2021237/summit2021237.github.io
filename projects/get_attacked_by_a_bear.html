Get Attacked By a Bear

This was my group's final project for an advanced database systems course. The project was to develop an information system with three different NoSQL systems to demonstrate polyglot persistence and use scalability and consistency features of those systems.

The instructor wanted the project to be creative and interesting to us, so we chose a system that recommends locations to get attacked by bears. From our problem statement,

Bear attacks are, despite common perception, quite rare. Even where bears are relatively common, they tend to either avoid humans or coexist peacefully, and many places have policies and strategies in place to prevent bear attacks. This poses a problem for anyone who wants the opportunity to be attacked by a bear. Even bear sightings can be hard to achieve in many places.

This project was written in Python and uses MongoDB, Neo4j, InfluxDB, and Kafka. The NoSQL systems were hosted on Debian virtual machines from the computer science department, and each of us could simultaneously run the information system, make changes to the data, and see each other's changes.

Features

  • Search for bear encounters by
    • Location
    • Person
    • Species
    • Wild or captive bear
    • Attack or sighting
    • Date
  • Track bear encounters (add, delete, edit, and view)
    • Location
    • Person
    • Species
    • Wild or captive bear
    • Attack or sighting
    • Date
  • Filter locations based on the likelihood of bear attacks or sightings
  • Recommend nearby locations with high bear attack frequencies
  • Display the frequency of bear attacks as time-series data over the years
  • Display times of the year that attacks occur more frequently
  • Recommend times for getting attacked utilizing time-series representation of historic data
  • Track bear population density for different locations to use for recommendations (add and edit)
  • Review bear encounters on a scale from 1 to 5 stars, with comments (add, delete, edit, and view)

Polyglot Persistence

Systems with polyglot persistence use multiple data storage systems depending on the ways the data will be used. Based on our features list, we decided to use MongoDB, Neo4j, and InfluxDB.

MongoDB

We track locations, encounters, and reviews of encounters, which all have many attributes. So it made since to use a system like MongoDB, which is document-based, and the ID of a document of one type can be a field in a document of another type, letting us simplify how locations are stored. The Encounter documents have a list of Reviews and reference a location ID, and the Location documents have a field containing encounter IDs of all the encounters that have occurred in that location.

Neo4j

Since we recommend locations with high bear attack frequencies, we use Neo4j. The graph database would also make it easier to add other methods for recommending locations in the future. Neo4j tracks the population densities of species in various locations, as well as encounters.

InfluxDB

We wanted to track encounters over time to determine the best times to try to get attacked by a bear. InfluxDB is a time-series database that stores measurements, a collection of points associating a values with times, and it has tags to add more information about the values stored. InfluxDB tracks the number of encounters for dates with tags providing more details on the encounters.

Scalability and Consistency

Another part of the project was learning about and implementing scalability and consistency features for the databases.

MongoDB has replica sets and shards, which can ensure eventual consistency, durability, and efficiency. We do not implement sharding because we do not have a lot of data. For consistency, we use a replica set with two data nodes and one arbiter node.

Neo4j has sharding and data federation for the Enterprise Edition. Since it would be outside the scope of this project to implement sharding (subgraph of the entire graph on each node) or data federation (disjoint graphs with different labels and relationship types on each node) ourselves, we do not use scalability and consistency options for the Neo4j part of the system.

InfluxDB automatically uses sharding with shard groups based on durations, but it has no additional scalability options for the free version. However, the paid versions allow multiple nodes and support clustering. Since we are using the free version, we run two separate InfluxDB nodes to support replication.

Recovery

For recovery, we use Kafka to queue events, and each database will consume events from their own queues. Then, each database ensures that it does not lose data when it goes down by reading all the missed modifications when it comes back up.
